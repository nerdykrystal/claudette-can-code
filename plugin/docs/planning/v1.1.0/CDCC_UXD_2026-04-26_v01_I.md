# CDCC v1.1.0 User Experience Document

> **Note on applicability.** CDCC has no GUI / web / native-window surface. Per PRD §6.4 and TRD §3.5, the UXD scope is limited to three non-visual surfaces: (1) CLI text output (stdout + stderr); (2) audit log entry format (sqlite rows + their JSON projection); (3) `cdcc explain` traceability matrix output. Visual sections of the v02 UXD template (color palette §2.1, typography §2.2, dark mode, etc.) are NA-justified throughout; interaction patterns, state catalog, and accessibility-as-delight criteria adapt cleanly to text surfaces.

## 1. Aesthetic Anchors

### 1.1 Reference Apps

### UXD-AR-01: gh CLI — Command Grammar + Stable JSON Output
What specifically: `gh <noun> <verb>` grammar; `--json` flag projects structured fields with documented stability; `gh status` and `gh pr view` give predictable layouts. URL: https://cli.github.com/. Does not apply: gh's auth, gh's editor invocation hand-off.

### UXD-AR-02: Stripe CLI — Pre-Action Communication + --dry-run
What specifically: every action prints a one-line summary of what's about to happen before doing it; `--dry-run` shows exactly what would change. URL: https://stripe.com/docs/cli. Does not apply: per-environment context switching is heavier than CDCC needs.

### UXD-AR-03: Cargo (Rust) — Phased Output + Last-Line-As-Verdict
What specifically: long-running commands emit phase-tagged progress lines (`Compiling`, `Finished`, `Running`); the LAST line is the verdict. URL: https://doc.rust-lang.org/cargo/. Does not apply: cargo's compile-graph is irrelevant.

### UXD-AR-04: AWS CloudTrail — Audit Entry Structured Fields
What specifically: every event has structured `who / what / when / source / result` fields; queryable; tamper-evident. Does not apply: cloud-native scope.

> **Stop & Verify past §1.1.** 4 reference apps named, each with what-specifically + does-not-apply. Surfaces (CLI grammar, pre-action messaging, phased output, audit entry shape) covered.

### 1.2 Brand Voice Expressed In Text Output

(Visual brand voice NA; "voice" here is text-output character.)

- **UXD-UD-voice-01**: Every CLI command emits ≤80-char first line summarizing what it's about to do before any FS write or git operation.
- **UXD-UD-voice-02**: Every audit-log entry's `decision / rationale / payload` triple is human-readable in `cdcc audit` default output (not raw JSON).
- **UXD-UD-voice-03**: Refusal messages name the rule + the offending state + the resolution path on three lines, in that order.
- **UXD-UD-voice-04**: Timestamps in CLI output are localized human-readable (`2026-04-26 12:33:23 UTC`); timestamps in audit DB are ISO 8601 with 'Z' suffix; never lex-compared (closes H-8).
- **UXD-UD-voice-05**: Exit codes are documented, stable, and listed in `cdcc help` (closes L-6).
- **UXD-UD-voice-06**: DRR pre-revert announcements (PRD-AR-NV-04) precede the revert with at least 100ms of delay so the user has time to register what's about to happen even on fast terminals.
- **UXD-UD-voice-07**: `cdcc explain` matrix uses fixed-width columns for terminal display + a `--json` flag for tooling.

### 1.3 Polish Criteria

### UXD-UP-01: Pre-Action Confirmation Always Present
- Observable: every CLI command emits a "About to do X" message to stderr before any FS write, sqlite write, or git operation.
- Test: pipe stderr; verify first line matches `^About to ` regex on every entry-point command.

### UXD-UP-02: Refusal Messages Are Tri-Layered (rule / state / resolution)
- Observable: H8 (A18 + roadmap P3 H8) refusal stderr has 3 distinct lines: rule_id + offending_path/persona + remediation step.
- Test: regex match against H8 stderr.

### UXD-UP-03: DRR Recovery Voice Is Pre-Revert + Plain English
- Observable: H9 emits `DRR firing on stage <id>: <one-line plain-English summary>; reverting to <hash>; re-delegating with: <one-line spec diff>` BEFORE `git reset --hard`.
- Test: integration test asserts stderr line precedence vs git operation.

### UXD-UP-04: Audit Output Aligns With ASCII Tableware
- Observable: `cdcc audit` default output uses fixed-width columns + colored severity tags (where TTY); `--json` strips formatting.
- Test: snapshot against fixture run.

## 3. Interaction Patterns

### 3.1 State Catalog (Per CLI Command Class)

CLI commands have these states: pre-action / acting / success / error. Each command defines:

### UXD-UN-generate-pre-action: `cdcc generate` Pre-Action State
- Output: `About to consume bundle at <abs-path>; will write plan.json + plan-state.json + install N hooks. Rerun with --dry-run to preview.` Pause 100ms before acting if TTY (per UXD-UD-voice-06 pattern).

### UXD-UN-generate-acting: Acting State
- Output: phased progress (`Parsing bundle...`, `Computing plan_hash...`, `Writing plan.json...`, `Installing hooks...`, `Audit entry recorded.`).

### UXD-UN-generate-success: Success State
- Output: `Plan written to <path>. <N> hooks installed. Audit entry <id> recorded. See cdcc audit --since=<ts> for details.`. Exit 0.

### UXD-UN-generate-error-bundle: Error — Bundle Invalid
- Output: 3-line refusal — rule + path + remediation. Exit 3 (per L-6 enum).

### UXD-UN-generate-error-pre-v0.3.0: Error — Bundle Methodology Stale
- Output: `Bundle is at methodology v0.1.x; auto-upgrading via /upgrade-bundle...` (then re-attempts). If shell-out unavailable: `Bundle methodology v0.1.x detected; v0.3.0 required. Run /upgrade-bundle on the bundle, then retry. See docs/upgrade-bundle.md.` Exit 4.

### UXD-UN-h8-refusal: H8 Protected-File Refusal State (A18 path-level + roadmap P3 H8)
- Output (stderr, 3 lines per UXD-UP-02):
  - `H8 REFUSED: protected_files.yaml rule for <persona>`
  - `path: <abs-path> (matched glob: <glob>)`
  - `resolution: edit protected_files.yaml to add <persona> to allowed_personas if intended; otherwise this refusal is correct.`
- Exit code: 2.
- (A22-aspect attribution corrected to A18 + roadmap P3 H8 per gate-48 NEW-FINDING-1 — mechanism unchanged.)

### UXD-UN-h9-drr-firing: H9 DRR Firing State (A21)
- Output: see UXD-UP-03 + UXD-UN-drr-recovery-success / failure below.

### UXD-UN-drr-recovery-success: DRR Recovery Success
- Output (post-revert + post-redelegation): `DRR recovered: stage <id> re-emitted within scope after revert to <hash>. Audit entry <recovery_event_id>.`. Exit 0.

### UXD-UN-drr-recovery-failure: DRR Recovery Failure (Catastrophic)
- See §3.2 catastrophic state.

### 3.2 Empty / Loading / Error / Success / Catastrophic States

### UXD-UN-empty-audit-query: `cdcc audit --since=<future-ts>` Empty State
- Output: `No audit entries match filters. Run cdcc audit --since=<install-ts> to see all entries.` Exit 0.

### UXD-UN-loading-audit-query: `cdcc audit` Loading State (1M-row DB)
- Output: streaming progress dots to stderr while iterating; rows to stdout. (Tested under TQCD-BG-02.)

### UXD-UN-error-bundle-parse: Bundle Parse Error State
- Output: 3-line refusal naming the parse-error site (file:line), the offending content, and the remediation.

### UXD-UN-success-explain: `cdcc explain` Success State
- Output: markdown table to stdout + audit entry. Exit 0.

### UXD-UN-catastrophic-drr-failure: DRR Failure (Recovery Cannot Complete)

**Trigger:** H9 fires but redelegation sub-agent fails N times in a row OR `git reset --hard` fails OR sub-agent spawn fails.

**Visual treatment (text):** stderr block, multi-line, severity tag `CATASTROPHIC` (red on TTY):
```
CATASTROPHIC: DRR cannot complete recovery for stage <id>.
violation_type: <type>
revert_target: <hash> (revert OK)
redelegation: failed <N> attempts
recovery_pass: false
audit_entry: <id>
DATA PRESERVATION: working tree was reverted to <hash>; nothing is destroyed.
NEXT STEP: review violation in cdcc audit --hook=H9 --decision=recovery_events; manually re-emit work or escalate.
```

**Copy:** above verbatim.

**Actions available:**
- `cdcc audit --hook=H9` to inspect.
- `git status` to confirm tree state.
- Manual user-driven re-emission.

**`data_preservation_strategy`:** working tree IS reverted to `revert_target` regardless of redelegation outcome (the revert is committed before the redelegation spawn). The `recovery_events:` audit entry persists in sqlite. No work is destroyed; the violation's emitted-then-reverted commit is reachable via `git reflog` for forensic inspection.

**`recovery_communication`:** when developer runs `cdcc audit --hook=H9` after the catastrophic, they see the full event chain. When developer runs the next `cdcc generate`, the CLI emits a one-time WARN: `Last DRR event at <ts> was catastrophic (id <recovery_event_id>); see audit log.`

### UXD-UN-catastrophic-audit-db-corrupted: Audit DB Corruption (Catastrophic)

**Trigger:** sqlite DB read fails integrity check at `cdcc audit` invocation.

**Visual treatment:** stderr block:
```
CATASTROPHIC: audit DB integrity check failed at <path>.
DATA PRESERVATION: DB file moved to <path>.corrupted-<ts>; new DB initialized.
NEXT STEP: run cdcc audit-replay <corrupted-path> to attempt recovery from WAL.
```

**`data_preservation_strategy`:** corrupted DB renamed (not deleted); WAL file preserved alongside. New DB starts fresh from current install state.

**`recovery_communication`:** `cdcc audit-replay` (deferred to v1.2.0; v1.1.0 ships with the renamed-not-deleted preservation only and a forward-pointer to v1.2.0 for the replay tool). UXD declares this NEXT STEP as deferred-but-named per `feedback_no_deferral_debt` discipline.

### 3.3 Animation + Transition Guidelines

NA — no GUI. Terminal output is line-by-line; pre-action delays per UXD-UD-voice-06 are the only "transition" timing.

### 3.4 Catastrophic Failure Voice

### UXD-UA-catastrophic-voice-01: Always Include Timestamp
- Per PRD-AR-NV-02. Audit entries always have ISO 8601 ts; CLI catastrophic blocks always include human-readable ts.

### UXD-UA-catastrophic-voice-02: Always Tell User What's Preserved
- Per UXD-UN-catastrophic-* `data_preservation_strategy:` field. Always names what survived.

### UXD-UA-catastrophic-voice-03: Never Use Raw Stack Traces In User-Facing Copy
- Stack traces go to audit DB `payload.stack_trace` field; CLI shows the human-readable summary.

### UXD-UA-catastrophic-voice-04: Always Offer At Least 2 Next-Actions
- Per UXD-UN-catastrophic-* — at minimum: an inspect action + an escalate-or-retry action.

### UXD-UA-catastrophic-voice-05: Never Silently Retry Without Telling User
- DRR redelegation announces each attempt. Bounded retry count (per TQCD-TC-S16 retry storm guard).

## 4. Information Architecture

### 4.1 Hierarchy Rules

For CLI output:
- **Primary**: the verdict (success / error / catastrophic). Always the LAST line per UXD-AR-03 cargo pattern.
- **Secondary**: phased progress.
- **Tertiary**: detailed diagnostics in audit DB (queryable, not echoed).

For `cdcc explain` matrix:
- Primary column: source ID.
- Secondary: source doc + section.
- Tertiary: test_path / reviewer_checklist_path.

### 4.2 Grouping And Prioritization

- Default `cdcc audit` output groups by hook_id then ts-descending.
- `cdcc explain` groups by source-doc (PRD → TRD → AVD → TQCD → UXD), then by ID.

### 4.3 Navigation Pattern

- CLI: `cdcc help` for command index; `cdcc <cmd> --help` for per-command help.
- Audit DB: `cdcc audit` with filter flags; rows can be pipe'd to `jq`.

## 5. Accessibility-As-Delight (Adapted)

### 5.1 ARIA-Equivalent Quality Criteria For CLI

NA in the literal ARIA sense (CLI). The text-equivalent: structured-output-format quality.

- Action verbs in command names (`generate`, not `gen`; `explain`, not `info`).
- Error copy describes outcome ("Bundle parse failed at PRD §1.4 line 45") not mechanism ("YAMLException: bad indent").
- Consistent vocabulary across commands (`plan_hash` always means the same thing).

### 5.2 Keyboard Navigation Quality (Terminal Equivalent)

NA — terminal is the developer's terminal emulator's responsibility. CDCC defers to user shell.

### 5.3 Screen Reader Experience For CLI

- Output structure is scannable: severity tag at line-start (`CRITICAL:`, `OK:`) so screen readers vocalize severity first.
- No ASCII art that loses meaning when read aloud.
- `--json` flag provides structured output for any tool that needs to parse (including assistive tech).

### 5.4 Sensory Preferences

- ANSI color is opt-in via TTY detection; piped output is plain text. NO_COLOR env honored.
- Verbose progress can be silenced via `--quiet` (deferred to v1.2.0 if non-trivial; v1.1.0 ships verbose-by-default).

## 7. Anti-Patterns To Avoid

### UXD-UA-AP-01: Cryptic Stack-Trace Errors
- Looks like: `TypeError: Cannot read property 'sections' of undefined at extractExcellenceSpec`. What it lacks: pointer to user action. Replacement: `Bundle TQCD missing required §5.2 traceability table. Add the table per TQCD template §5.2 and retry. (debug: src/core/plan-generator/index.ts:23 see audit log)`.

### UXD-UA-AP-02: Silent Success
- Looks like: `cdcc generate` exits 0 with no output. Lacks: confirmation. Replacement: per UXD-UN-generate-success.

### UXD-UA-AP-03: ANSI Spam In Piped Output
- Looks like: `cdcc audit | jq` returns escape codes. Replacement: TTY-detection on stdout; auto-degrade.

### UXD-UA-AP-04: Refusal Without Resolution
- Looks like: `H8 refused.` (one line, no path, no fix). Replacement: per UXD-UN-h8-refusal three-line format.

### UXD-UA-AP-05: Catastrophic With No Data-Preservation Statement
- Looks like: catastrophic block without naming what survived. Replacement: per UXD-UA-catastrophic-voice-02.
