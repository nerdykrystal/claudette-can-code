# CDCC v1.1.0 Product Requirements Document

## 1. Product Identity `[independent]`

### 1.1 Product Name

**Claudette-Can-Code (CDCC)** — a Claude Code plugin in the Martinez Methods ASAE stack family (alongside the `/asae` skill, `/dare-to-rise-code-plan` skill, and the canonical `commit-msg` hook chain).

### 1.2 Version

**v1.1.0** (remediation release; product identity unchanged from v1.0.4 baseline). Two version streams remain: `package.json` semver (advances v1.0.4 → v1.1.0) and `plugin.json` plugin-protocol version (advances v0.1.0 → v0.2.0). M-8 version-skew finding is partially closed by documenting the two streams' distinct purposes in this PRD and in the README; full reconciliation is a v2.0 candidate.

### 1.3 One-Line Description

**Non-bypassable D2R plan enforcement for Claude Code, with byte-equal audit logs, structural Detect-Revert-Redelegate recovery on F-class violations, and capability-bounded protected-file enforcement.**

### 1.4 Non-Visual Excellence Anchors `[parallelizable-with §2]`

CDCC's surfaces are all non-visual: CLI output, append-only audit log files, the `cdcc explain` traceability matrix, `protected_files.yaml`, hook installer effects on `.claude/settings.json`. The §1.4 anchors prevent the implementer from defaulting to "generic CLI plugin equivalent of bland React-component-library output."

#### 1.4.1 Operational Ergonomics Anchors

### PRD-AR-01: gh CLI — Verb-Noun Grammar + Structured Output
What specifically: consistent `gh <noun> <verb>` command shape; stable `--json` output with documented fields; stable exit codes. URL: https://cli.github.com/. Does not apply: gh's auth model is GitHub-specific; CDCC has no auth surface.

### PRD-AR-02: Stripe CLI — Pre-Action Communication & --dry-run
What specifically: the CLI prints what it's about to do before doing it; `--dry-run` is structural, not optional. URL: https://stripe.com/docs/cli. Does not apply: Stripe's per-environment context (test/live) is heavier than CDCC needs.

#### 1.4.2 Failure-Communication Anchors

### PRD-AR-03: Cloudflare Status Page — Cause + Impact + Scope + ETA + Workaround
What specifically: every incident report names cause + impact + scope + ETA + workaround. URL: https://www.cloudflarestatus.com/. Does not apply: multi-region complexity is N/A.

### PRD-AR-04: Linear — Error States Suggest Concrete Next Action
What specifically: error states explain what's wrong AND what the user can do; remediation suggestion is structural not optional. Does not apply: Linear's UI surfacing layer; CDCC translates the same discipline to CLI exit messages and audit-log entries.

#### 1.4.3 Audit-Trail Rendering Anchors

### PRD-AR-05: AWS CloudTrail — Structured Event Log
What specifically: structured who/what/when/result/payload fields per event; queryable; tamper-evident. Does not apply: cloud-native scope is N/A; CDCC's audit log is local-file-on-developer-machine.

### PRD-AR-06: git — Commit Message As Load-Bearing
What specifically: every change has a message; the message is enforced as load-bearing communication. Does not apply: nothing — CDCC inherits this discipline directly from the canonical `commit-msg` hook chain.

#### 1.4.4 Documentation Excellence Anchors

### PRD-AR-07: React Docs (Post-2023 Rewrite) — Concept-First
What specifically: concept-first, then API. URL: https://react.dev/. Does not apply: React's audience is broader.

### PRD-AR-08: Stripe API Docs — Side-By-Side Examples
What specifically: every endpoint has runnable examples in multiple languages side-by-side. Does not apply: CDCC has only one runtime (Node CLI).

#### 1.4.5 Non-Visual Brand Voice — Concrete Decisions

- **PRD-AR-NV-01**: Every CLI command prints what it's about to do before doing it. `--dry-run` is structural for `cdcc generate` and `cdcc consume`.
- **PRD-AR-NV-02**: Every audit-log entry has the triple `decision / rationale / payload`, never `event / timestamp` only. Schema-validated, AJV-strict.
- **PRD-AR-NV-03**: Every error message includes a remediation suggestion with a concrete next step (e.g., "run `cdcc audit --since=2026-04-26 --hook=H9` to see the recovery_events that fired"), not just an error name.
- **PRD-AR-NV-04**: Every recovery (A21 DRR fire) is announced before the revert, not after. The user sees "DRR firing on stage `<stage_id>` violation `<violation_type>`; reverting to `<commit-hash>` and re-delegating with diff `<spec_diff>`" before the revert lands.
- **PRD-AR-NV-05**: Every protected-file violation refusal (H8 / A18 + roadmap P3 H8) names the rule, the protected path, the persona, and the resolution path ("edit `protected_files.yaml` to add `<persona>` to `allowed_personas` for `<path>` if this is intended; otherwise this refusal is correct").
- **PRD-AR-NV-06**: Exit codes are documented in `docs/exit-codes.md` with a stable enum (closes L-6).
- **PRD-AR-NV-07**: Audit log entries are byte-equal across crashes — sqlite WAL durability replaces JSONL fsync (closes C-2).

#### 1.4.6 Non-Visual Anti-Patterns

### PRD-AR-AP-01: Silent Hook Refusal
What it looks like: a hook refuses a tool call by exit-code 2 with no stderr message. What it lacks: any pointer to which rule fired or how to fix. Replacement: every refusal prints `HOOK <id> REFUSED: <rule-name> — <one-line-explanation>; remediation: <next-step>` to stderr (per PRD-AR-NV-03 / PRD-AR-NV-05).

### PRD-AR-AP-02: Audit Log "Just Works" Until It Doesn't
What it looks like: serial-write happy-path tests passing, no concurrent-writer or fault-injection tests. What it lacks: evidence the byte-equal-across-crashes claim is true. Replacement: sqlite WAL with ACID guarantees + concurrent-writer fault-injection tests + crash-recovery tests with `kill -9` mid-write (per PRD-AR-NV-07; closes C-2 + M-4).

### PRD-AR-AP-03: Fail-Open On Unknown State
What it looks like: H4 returning exitCode 0 (allow) when stage-not-found. What it lacks: enforcement. Replacement: every hook fails CLOSED on missing/unknown state with a documented refusal message naming the resolution path (closes C-3).

> **Stop & Verify before continuing past §1.4.** Confirmed: 4 reference categories with ≥2 anchors each; 7 concrete brand voice decisions; 3 named anti-patterns each with concrete replacement. Bundle is ready for §2.

- C-3: H4 fails OPEN on stage-not-found, bypassing the "non-bypassable" claim.

Evidence: gate-22 audit log (29 findings, rater CONFIRMED via real subagent spawn `agentId a8bab1a8cc4677a7b`); CCC build empirical record of F8/DRR pattern firing 4× in a single MVP build (per Two-Axis Pitch §2).

## 4. User Journeys `[requires §2]`

### PRD-UJ-01: Practitioner Generates A Plan From A 5-Doc Bundle
- Goal: take an approved 5-doc bundle and produce a Claude Code plan with all hooks installed and a baseline audit log entry.
- Steps: practitioner runs `cdcc generate <planning-dir>`; CDCC reads BIDX-1 Methodology field; if v0.3.0 5-doc, parses PRD/TRD/AVD/TQCD/UXD; computes plan ID via stable hash with separator (closes M-10); writes plan.json + plan-state.json with HMAC integrity (closes H-6); installs hooks atomically with backup (closes H-1, M-3); emits `plan_generated` audit entry to sqlite (closes C-2); prints "Plan written to <path>; <N> hooks installed; audit log entry <id> recorded."
- Outcome: plan + hook-installed-settings + plan-state + audit entry, all atomic.
- Pain points eliminated: Silent CLI success (PRD-AR-AP-02-equivalent at v1.0.4); fail-open on stage-not-found (C-3); plan-state-never-written (H-6).

### PRD-UJ-02: Sub-Agent Violation Triggers DRR Auto-Recovery
- Goal: when a sub-agent's emitted work product violates scope (modifies protected file, fabricates a commit, drops coverage below threshold), CDCC reverts the work and re-delegates with corrected constraints — no human in the loop.
- Steps: sub-agent emits Write/Edit/Bash that mutates `protected_files.yaml`-listed path; H8 PreToolUse refuses (A22). If somehow it slips (e.g., Bash heredoc): parent verification step fires; H9 PostToolUse detects scope violation; emits `recovery_events:` entry naming violation_type + detected_by + revert_target; reverts working tree to revert_target; spawns re-delegation sub-agent with `redelegation_spec_diff` constraints; verifies `recovery_pass: true`. User-visible message: "DRR firing on stage `<id>` violation `<type>`; reverting to `<hash>` and re-delegating with corrected scope. See `cdcc audit --since=<ts> --hook=H9` for details" (per PRD-AR-NV-04).
- Outcome: scope violation reverted, work re-emitted within scope, audit log records the recovery.
- Pain points eliminated: LLM fabrication shipping to disk (Stage 07 of CCC build); silent partial-revert with no audit trail.

### PRD-UJ-03: Practitioner Runs `cdcc explain <plan-id>` For Traceability
- Goal: surface the chain `PRD-FR → TRD-NFR → AVD-AC → TQCD-EC → test_path` for any plan-id.
- Steps: `cdcc explain <plan-id>` reads plan.json + bundle BIDX-3 heading-prefix table; emits a markdown table to stdout; pipes to `--json` for tooling.
- Outcome: practitioner can audit their own bundle's traceability; surfaces orphan headings (per BIDX integrity rule).
- Pain points eliminated: "where is FR-04 tested?" requiring grep across four files.
