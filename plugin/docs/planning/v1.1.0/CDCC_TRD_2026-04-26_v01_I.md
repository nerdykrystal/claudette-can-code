## 2. Functional Requirements `[requires PRD §2, §3, §4]`

### 2.1 Core Functional Requirements

### TRD-FR-01: System MUST Extract Excellence Spec From Bundle Content (closes C-1, M-2)
- Statement: `extractExcellenceSpec(bundle: ParsedBundle): ExcellenceSpec` MUST take a parsed bundle as argument and return spec content extracted from `bundle.tqcd.sections` and `bundle.avd.sections`. MUST NOT return hardcoded strings.
- Rationale: implements PRD-SO-01 + PRD-AR-NV-04; closes the F13-violation-by-the-F13-prevention-plugin defect.
- Acceptance: unit test feeds a fixture bundle with TQCD §5.2 traceability table + AVD §7 ADRs; asserts return value contains both. Negative test: empty bundle raises a typed error, not silent success. Function name renamed if needed (`extractExcellenceSpec` keeps name; behavior brought to match name).
- Priority: Must-Have.

### TRD-FR-02: Audit Log MUST Be Byte-Equal Across Crashes (closes C-2, M-1, M-4)
- Statement: every committed audit-log entry MUST survive `kill -9` of the writer process at any point in the entry's emission, with no partial / torn / duplicate entries.
- Rationale: implements PRD-SO-05 + PRD-AR-NV-07; backs "non-bypassable" claim with crash durability.
- Acceptance: fault-injection test runs 10 concurrent writers each emitting 1000 entries while a parallel SIGKILL fires every 50ms; post-crash query returns N entries where N ≤ 10000, every returned entry passes schema validation, no duplicate entry IDs, no torn rows. Run on Windows + macOS + Linux.
- Priority: Must-Have.

### TRD-FR-03: H4 Hook MUST Fail CLOSED On Stage-Not-Found (closes C-3)
- Statement: H4 model-assignment hook MUST return exitCode 2 (refuse) with structured stderr message when `currentStage` cannot be resolved from plan-state.json or when plan-state.json itself is missing/malformed. MUST NOT return exitCode 0 (allow) on unknown state.
- Rationale: implements PRD-AR-AP-03 replacement; closes "non-bypassable" bypass.
- Acceptance: integration test invokes the tool while plan-state.json is (a) missing, (b) malformed, (c) present-but-stage-id-unrecognized; all three cases return exit 2 with documented stderr message.
- Priority: Must-Have.

### TRD-FR-04: `cdcc generate` MUST Write plan-state.json With HMAC Integrity (closes H-6)
- Statement: at successful generate completion, plan-state.json MUST be written atomically (tmp+fsync+rename+fsync(parent)) at the path H4 reads from, with an HMAC-SHA256 trailer keyed by a per-install secret (rotated per-install, stored at `~/.claude/plugins/cdcc/.hmac-key` with 0600 perms).
- Rationale: implements PRD-SO-02 partially; closes the "plan-state read but never written" anomaly.
- Acceptance: test verifies file exists post-generate, HMAC validates, kill -9 mid-write leaves either empty path (treated as missing → fail closed per FR-03) or valid HMAC'd file (no torn writes).
- Priority: Must-Have.

### TRD-FR-05: Hook ID Single Source Of Truth In plugin.json (closes H-3)
- Statement: hook IDs + handler paths + event types MUST be declared once in plugin.json. CLI generate path MUST read plugin.json at runtime; MUST NOT hardcode the entries array.
- Rationale: closes H-3 dual-source defect.
- Acceptance: cross-validation test reads plugin.json, asserts hook count + IDs + paths match what handleGenerate registers.
- Priority: Must-Have.

### TRD-FR-06: System MUST Auto-Invoke /upgrade-bundle On Pre-v0.3.0 Bundles (closes M-9)
- Statement: `cdcc consume <planning-dir>` MUST inspect BIDX-1 Methodology field; if methodology rollup version < 0.3.0, MUST shell out to `/upgrade-bundle` skill (or print a remediation message linking to it if shell-out path is unavailable in the runtime); refuse to consume future-version-unknown bundles.
- Rationale: implements PRD-SO-04.
- Acceptance: integration test fixtures: (a) v0.1.x 4-doc, (b) v0.3.0 5-doc, (c) v0.4-future. (a) triggers /upgrade-bundle, (b) proceeds, (c) refused with remediation message.
- Priority: Must-Have.

### TRD-FR-07: H8 PreToolUse Hook MUST Refuse Protected-File Mutations (implements A18 path-level + roadmap P3 H8 Capability-Scope Runtime Enforcer)
- Statement: H8 MUST read `protected_files.yaml` at session start; on every Write/Edit/Bash tool call, MUST match the target path against listed globs; if matched AND current persona role-manifest is not in `allowed_personas` for that glob, MUST return exitCode 2 with stderr message naming the rule + path + persona + resolution.
- Rationale: implements PRD-SO-03 (per gate-48 NEW-FINDING-1 attribution correction). H8 is the Capability-Scope Runtime Enforcer per Multi-Taxonomy FM Scoreboard v03 product roadmap P3, operationalizing A18 (Capability-Scope Attestation) at execution time, not just at gate time. **A22 standalone-aspect codification REJECTED by FM thread rater 2026-04-26 (failed ≥3-FM-per-aspect guard); path-level concern folded into A18 per Two-Axis Pitch v02 §3.**
- Acceptance: per-tool unit tests for Write/Edit/Bash/heredoc-via-Bash; persona-not-in-list refusal verified; persona-in-list pass-through verified; missing protected_files.yaml treated as "no protections" (with WARN log) per least-surprise principle.
- Priority: Must-Have. (Not feature-flagged: H8 is a roadmap P3 item with prior approval per v03 scoreboard, distinct from H9 DRR which remains experimental-flagged.)

### TRD-FR-08: H9 PostToolUse Hook MUST Detect-Revert-Redelegate On Parent Verification Violations (codifies A21)
- Statement: H9 MUST fire on parent-verification-completion event (Stop hook variant); when parent verification reports a violation_type from {scope_violation, false_attestation, coverage_drop, protected_file_mutation, role_boundary, fabrication}, MUST emit a `recovery_events:` audit entry with all fields populated (stage_id, violation_type, detected_by, revert_target, redelegation_spec_diff, recovery_pass), revert working tree to revert_target via `git reset --hard <hash>`, spawn re-delegation sub-agent with redelegation_spec_diff payload, and verify recovery_pass at completion. **Status: PROPOSED PENDING FM THREAD RATER VERIFICATION.**
- Rationale: implements PRD-SO-02; codifies A21.
- Acceptance: end-to-end test simulates each violation_type and asserts each step in the chain. Failure-to-redelegate path emits `recovery_events:` with `recovery_pass: false` and surfaces to user without retry-loop.
- Priority: Must-Have. **Feature-flagged behind `cdcc.experimental.drr`.**

### TRD-FR-09: H6 MUST Combine Step-History + Cost-Telemetry (per merge decision)
- Statement: H6 PreToolUse on Write/Edit/Bash MUST emit a single audit entry combining step-history fields (step_id, hash_of_inputs) and cost-telemetry fields (model, input_tokens_est, output_tokens_est) when Claude Cost dependency is loaded. If Claude Cost is unavailable, step-history fields emit alone (no failure).
- Rationale: implements PRD-OQ-01 provisional answer (merged H6).
- Acceptance: tests cover (a) Claude Cost present, both fields populated; (b) Claude Cost absent, step-history populated, cost fields omitted; (c) Claude Cost throws, error logged but step-history still emits.
- Priority: Must-Have.

### TRD-FR-10: `cdcc explain <plan-id>` MUST Emit Traceability Matrix (PRD-UJ-03)
- Statement: command reads plan.json + bundle BIDX-3; emits markdown table with columns `source_id | source_doc | trd_section | avd_component | tqcd_test | test_path`; `--json` flag produces equivalent JSON.
- Acceptance: e2e test against the bundle authored by this very release.
- Priority: Must-Have.

### TRD-FR-11: `cdcc generate` MUST Backup settings.json Before Atomic Write (closes M-3)
- Statement: backup as `.claude/settings.json.cdcc-bak-<ISO8601>` before applying hook-installer changes; backup retained until next successful generate.
- Acceptance: test verifies backup exists, can be restored manually via `cdcc rollback`.
- Priority: Must-Have.

### TRD-FR-12: Audit Query MUST Stream, Not readFileSync The World (closes H-5)
- Statement: `cdcc audit` query MUST stream rows from sqlite via prepared-statement iteration with --since filter pushed to SQL `WHERE`. Memory budget per query bounded by row-count batch (≤1000 rows in flight).
- Acceptance: perf test on a 1M-row audit DB asserts RSS-after-query stays within 50MB ceiling.
- Priority: Must-Have.

### TRD-FR-13: Audit Log Filename Discipline MUST Use Server Time, Not Entry ts (closes H-7, H-8)
- Statement: audit DB partitioned by server-clock day (sqlite attached DB per day) OR single-DB-with-day-index — either way, entry placement does not depend on caller-emitted `ts` field. ISO 8601 query filters use proper Date parsing, not lex compare.
- Acceptance: test feeds entries with timezone-shifted ts; verifies they land in the correct server-clock day; query with `+00:00` and `Z` mixed forms returns identical results.
- Priority: Must-Have.

### TRD-FR-14: Bundle Hash MUST Use Separator (closes M-10)
- Statement: plan ID hash MUST use a non-empty separator between concatenated documents (e.g., `\x1f` US character or `\n---\n`). MUST NOT be raw concatenation.
- Acceptance: collision-resistance test feeds two bundles where naive concat would collide; asserts plan IDs differ.
- Priority: Must-Have.

### TRD-FR-15: Plan ID MUST Be Either Real RFC 4122 UUID Or Renamed To `plan_hash` (closes L-3)
- Statement: rename to `plan_hash` (16-char hex prefix of sha256). Drop the fake-UUID shape.
- Acceptance: schema lint verifies field name.
- Priority: Should-Have.

### TRD-FR-16: CLI MUST Support `--force` On Generate (closes L-5)
- Statement: re-run of `cdcc generate` against existing plan refuses unless `--force` is supplied; refusal stderr names the path + last-write timestamp + the resolution (`--force` to overwrite, `cdcc rollback` to revert).
- Acceptance: integration test.
- Priority: Should-Have.

### TRD-FR-17: CLI Exit Codes MUST Be Documented (closes L-6)
- Statement: enum at `src/cli/exit-codes.ts` + `docs/exit-codes.md`. Every exit() call references the enum.
- Acceptance: lint rule enforces no inline numeric exit codes.
- Priority: Should-Have.

## 3. Non-Functional Requirements `[requires §2]`

### 3.1 Performance & Scale Requirements

### TRD-NFR-3.1-01: `cdcc generate` Wall-Clock
- p50 ≤ 500ms, p95 ≤ 1500ms, p99 ≤ 3000ms on a 5-doc bundle ≤ 200KB total.

### TRD-NFR-3.1-02: `cdcc audit --since` Wall-Clock + Memory
- Query against 1M-row DB: p95 ≤ 2s, RSS ceiling 50MB. (Closes H-5.)

### TRD-NFR-3.1-03: Hook Latency Budget
- Per-tool hook (H1-H9) p95 wall-clock ≤ 50ms; p99 ≤ 200ms. Slow hooks degrade developer experience disproportionately because they fire per Claude Code tool call.

### TRD-NFR-3.1-04: Performance Budget Allocation
- 60% generate (bundle parsing + plan write + settings install), 30% audit query, 10% hooks. Total 100%.

### TRD-NFR-3.1-05: Load Profile
- Steady (developer interactive) — 1 concurrent generate + ~10–50 concurrent hooks during a Claude Code session. No bursty pattern. Source: CCC build calibration.

> **Stop & Verify past §3.1.** All numbers specific (ms, MB, RSS); budget allocation totals 100%; load profile sourced.

### 3.2 Reliability & Resilience Requirements

### TRD-NFR-3.2-01: Audit Log Durability
- Byte-equal across crashes (per FR-02). RTO: instant (next process can read DB immediately). RPO: zero (ACID committed entries survive).

### TRD-NFR-3.2-02: Hook Failure Mode
- Every hook fails CLOSED (refuse with documented exit code) on missing/malformed state. Closes C-3.

### TRD-NFR-3.2-03: Resilience Patterns
- Retries: NONE inside hooks (would amplify violation latency; parent verification + DRR is the recovery pattern).
- Circuit breakers: NA (no remote dependencies).
- Idempotency: `cdcc generate` is idempotent given same bundle hash + `--force`; refused otherwise.
- Timeouts: each hook ≤ 5s wall before forced kill (per Claude Code platform default).

### TRD-NFR-3.2-04: Disaster Recovery
- DR is `git reset --hard` of working tree (the local repo IS the durable store; sqlite audit DB rebuilds from event replay if corrupted, with last-N-events potentially lost). DR runbook deferred to RUNBOOK-recovery in v1.2.0.

### 3.3 Security Requirements

### TRD-NFR-3.3-01: No Auth Surface
- CDCC has no network surface, no remote auth. Track 15 NA.

### TRD-NFR-3.3-02: Secret Hygiene
- HMAC key (per FR-04) stored at `~/.claude/plugins/cdcc/.hmac-key` with 0600 perms; not committed; gitignored at install time; rotated per-install (new install = new key).

### TRD-NFR-3.3-03: No Secrets In Audit Log
- Audit-log entries scrub fields named `password`, `token`, `key`, `secret`, `auth*` per regex; replaced with `<redacted>`. Test verifies.

### TRD-NFR-3.3-04: Standards
- OWASP ASVS Level 1 (developer-tool-tier minimum); OWASP LLM Top 10 LLM06 (excessive agency) — A21 + A18 + roadmap P3 H8 (the corrected attribution post-A22-rejection) are the primary mitigations. CERT secure coding for TypeScript/Node.

### TRD-NFR-3.3-05: Threat Model Reference
- Track 9 threat model: STRIDE applied. Top threats: (T1) sub-agent fabrication committing to disk → mitigated by H9 + parent verification; (T2) protected-file mutation → mitigated by H8; (T3) audit-log tampering → mitigated by sqlite WAL + per-entry HMAC trailer (deferred to v1.2.0; v1.1.0 ships sqlite without per-entry HMAC).

### 3.4 Privacy Requirements

### TRD-NFR-3.4-01: PII Minimization
- Audit log captures user prompts incidentally via Claude Code event payloads; redaction rules per NFR-3.3-03.
- No collection beyond what Claude Code itself emits.
- Track 13 NA (not regulated; not external).

### 3.5 Accessibility Requirements

NA — no UI surface. WCAG 2.1 AA inapplicable to CLI text output beyond "use stable, parseable output formats" (covered by NFR-3.8 below).

### 3.6 Maintainability Requirements

### TRD-NFR-3.6-01: Codebase Lifespan
- 24-month planned lifespan for v1.x; v2.0 may rewrite.

### TRD-NFR-3.6-02: Code Quality Gates
- TypeScript strict; no `any` outside documented AJV-binding sites (closes L-2 by reducing the count from 6 to ≤2 with documented exemption).
- Cyclomatic complexity median <8, p95 <12.
- 100% line + branch coverage on new v1.1.0 code (per D2R hardwire).

### 3.7 Portability Requirements

### TRD-NFR-3.7-01: Target Platforms
- Windows 11 (Powershell + Git Bash), macOS 14+, Linux Ubuntu 22.04+.
- Node ≥ 18 (LTS gate).
- better-sqlite3 native binding pre-built for tier-1 platforms.

### 3.8 Observability Requirements

### TRD-NFR-3.8-01: Audit Log IS The Observability Surface
- Structured: schema-validated, AJV-strict (per FR-13 schema rev includes recovery_events / attempted_violations / cost_telemetry).
- Queryable: `cdcc audit` with --since, --hook, --decision filters.
- Format: sqlite WAL DB at `~/.claude/plugins/cdcc/audit.sqlite`; each row JSON-serializable for tooling.

### TRD-NFR-3.8-02: SLI / SLO
- SLI: hook latency p95 (per NFR-3.1-03). SLO: ≥ 99% of hook invocations under p95 budget per developer-day.
- No remote dashboards (local tool); developers query their own audit log.

## 4. Integration Requirements `[requires §2]`

### TRD-INT-01: Claude Code Hook Protocol
- Purpose: register H1-H9 hooks with Claude Code.
- Protocol: Claude Code plugin manifest at `plugin.json`; events PreToolUse / PostToolUse / UserPromptSubmit / Stop; structured exit codes.
- Auth: NA (in-process plugin, no remote).
- Rate limits: platform-imposed per-tool latency budget (≤5s default).
- Fallback: if Claude Code platform protocol breaks, plugin-level error logged to audit DB; user sees CLI message.

### TRD-INT-02: Better-SQLite3 Native Binding
- Purpose: WAL-mode audit DB.
- Protocol: synchronous Node API.
- Fallback: NONE — sqlite is required; install fails on platforms without native binding (Windows ARM may need pre-built binary).

### TRD-INT-03: @martinez-methods/claude-cost (npm dependency)
- Purpose: cost-telemetry input for H6.
- Protocol: in-process JS API: `costFor(toolUse: ToolUseEvent): CostEstimate`.
- Fallback: try/catch around CC call; on failure, H6 emits step-history without cost fields (per FR-09).

### TRD-INT-04: Git CLI (PATH dependency)
- Purpose: H9 DRR uses `git reset --hard` and `git log` for revert_target resolution.
- Fallback: H9 refuses to fire if `git` not on PATH; emits `recovery_events:` with `recovery_pass: false, reason: "git_unavailable"`.

### TRD-INT-Source-01: protected_files.yaml (repo-level)
- Update frequency: on user edit.
- Format: YAML; schema validated at session start.
- Authority: repo-owner.
- Staleness tolerance: read on session-start; H8 refuses on parse error.
