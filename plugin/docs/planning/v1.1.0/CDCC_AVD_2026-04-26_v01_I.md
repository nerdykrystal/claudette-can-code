## 2. System Shape

### 2.1 One-Paragraph System Description

CDCC is a Claude Code plugin distributed as an npm package. It executes in three modes: (a) one-shot CLI commands (`cdcc generate / audit / dry-run / explain / rollback / migrate-audit-log`); (b) per-tool-call hooks loaded into Claude Code's plugin runtime (H1-H9, in-process); (c) a per-developer-machine append-only audit store (sqlite WAL DB at `~/.claude/plugins/cdcc/audit.sqlite`). It is local-only — no network, no remote services, no shared state across machines.

### 2.2 Architectural Style

- **Plugin + CLI hybrid.** Hook handlers and CLI commands share a core library.
- **Synchronous hot path for hooks.** Per-tool latency budget (≤50ms p95) forbids async I/O on the hook hot path beyond the sqlite write.
- **Stateful (sqlite + plan-state.json + audit.sqlite + protected_files.yaml) on a single machine.**
- **Locally executed; no client/server split.**

### 2.3 Surface Layers

- **CLI** — human developer + Claude Code agent.
- **Hooks** — Claude Code plugin runtime (in-process JS).
- **Audit DB** — sqlite consumed by `cdcc audit` and external developer tooling.
- **Config files** — `protected_files.yaml`, `~/.claude/settings.json` (mutated by hook installer).

All four surfaces are wrappers around the core library at `src/core/`. The CLI and hooks are thin entry points.

## 4. Data Flow

### AVD-DF-01: Practitioner Generates A Plan (PRD-UJ-01)
- Trigger: `cdcc generate <planning-dir>`.
- Steps: argv → CLI Entry → Bundle Parser (load BIDX-1; if methodology < v0.3.0, shell out to /upgrade-bundle then re-load) → Plan Generator (extractExcellenceSpec(bundle); compute plan_hash with separator) → Plan-State Store (write plan-state.json + HMAC) → Hook Installer (read plugin.json; merge settings.json with backup) → Audit Logger (emit `plan_generated` row).
- Output: plan.json + plan-state.json + settings.json updated + audit row + stderr confirmation per PRD-AR-NV-01.
- Latency: per TRD-NFR-3.1-01.

### AVD-DF-02: Tool-Call Hook Chain (per Claude Code event)
- Trigger: Claude Code emits PreToolUse / PostToolUse / UserPromptSubmit / Stop event.
- Steps: Claude Code runtime → relevant hook handler (H1-H9 by event registration) → Audit Logger (emit hook-specific row) → return exitCode 0/2 to Claude Code.
- Output: row in audit.sqlite + exit code.
- Latency: per TRD-NFR-3.1-03.

### AVD-DF-03: H8 Protected-File Refusal (A18 + Roadmap P3 H8) (PRD-UJ-02 prevention path)
- Trigger: PreToolUse on Write/Edit/Bash with target path matching protected_files.yaml glob.
- Steps: H8 → Protected Files Resolver.match(path, persona) → if not allowed, emit `attempted_violations` row + return exitCode 2 with stderr per PRD-AR-NV-05.
- Output: refusal + audit row.

### AVD-DF-04: A21 DRR Recovery (PRD-UJ-02 recovery path)
- Trigger: parent verification step reports violation_type.
- Steps: H9 → Recovery Verifier → emit `recovery_events` row (pre-revert) → `git reset --hard <revert_target>` → spawn re-delegation sub-agent with redelegation_spec_diff → on completion, emit `recovery_events` row (post) with recovery_pass.
- Output: working tree reverted + sub-agent re-emit + 2 audit rows.

### AVD-DF-05: Audit Query (`cdcc audit`)
- Trigger: `cdcc audit --since=<ts> --hook=<id>`.
- Steps: CLI Entry → Audit Logger.query() → sqlite prepared statement with filter pushed to WHERE → batch iterator (≤1000 rows in flight) → stdout JSON / table.

### AVD-DF-06: `cdcc explain` Traceability (PRD-UJ-03)
- Trigger: `cdcc explain <plan-id>`.
- Steps: CLI → load plan.json + parse BIDX-3 from bundle → cross-walk PRD-FR ↔ TRD-NFR ↔ AVD-AC ↔ TQCD-EC ↔ test_path → render markdown or JSON.

## 7. Architectural Decisions (Mini-ADRs)

### AVD-AD-01: Audit Log JSONL → SQLite WAL (closes C-2)

**Status:** Accepted.

**Context:** v1.0.4 JSONL daily files use open(a)+write+fsync. POSIX guarantees atomic appends only for writes <PIPE_BUF (4096B); larger payloads can interleave under concurrent invocations. Windows has no PIPE_BUF guarantee at any size. The "byte-equal append-only audit log" claim is structurally false under concurrency.

**Options Considered:**
- SQLite WAL via better-sqlite3 (chosen).
- File-per-entry + atomic-rename pattern (rejected — unbounded inode pressure; complicates query).
- POSIX advisory locks + JSONL (rejected — Windows lock semantics differ; complexity).
- LMDB / RocksDB (rejected — adds heavyweight dep for the same ACID guarantees sqlite already provides).

**Decision:** Use sqlite in WAL mode for audit log durability and concurrency.

**Rationale:** ACID; battle-tested; cross-platform via better-sqlite3 native binding; query-friendly for `cdcc audit`; supports streaming via prepared statements (closes H-5 simultaneously).

**Consequences:** Native dependency on every install (better-sqlite3 build); migration tool needed for v1.0.x JSONL→sqlite (`cdcc migrate-audit-log`); schema versioning becomes a real concern (closes L-7 for v1.1.0; future schemas need migrations).

**Reversal Cost:** **Heavy.** Reverting to JSONL would require: (a) re-implementing concurrency safety (circular problem); (b) updating `cdcc audit` query to JSONL streaming; (c) rewriting migration tool reverse-direction. Estimated 2-3 weeks. Justification for accepting: sqlite is the mainstream choice for local-app durability; the alternative (claiming JSONL works when it doesn't) is the worse path.

### AVD-AD-02: plan-state.json Written At Generate Completion + HMAC Integrity (closes H-6)

**Status:** Accepted.

**Context:** v1.0.4 H4 reads plan-state.json but no code in the codebase WRITES it (rater CONFIRMED). H4 fails closed on missing file (catches first-run); but stage-not-found path within an extant file fails OPEN (C-3).

**Options Considered:**
- Write at generate completion + HMAC integrity (chosen).
- Write at every hook fire (rejected — unnecessary write amplification).
- Drop plan-state entirely; H4 reads from plan.json (rejected — plan.json is bundle-derived; plan-state captures runtime stage progression which is distinct).

**Decision:** Write plan-state.json atomically at the end of `cdcc generate`. Sign with HMAC-SHA256 trailer keyed by per-install secret.

**Rationale:** Closes H-6 cleanly. HMAC defends against developer-tampered plan-state advancing the stage past gate boundaries.

**Consequences:** HMAC key file management (rotation, backup, recovery). Failed verify → H4 fails CLOSED with remediation message.

**Reversal Cost:** **Moderate.** ~3 files (plan-state writer, HMAC verifier, H4 read path). 1-3 days.

### AVD-AD-03: Hook IDs — Single Source Of Truth In plugin.json (closes H-3)

**Status:** Accepted.

**Context:** v1.0.4 hardcodes hook entries in cli/index.ts AND declares them in plugin.json. Drift is silent.

**Options Considered:**
- Single source in plugin.json read at runtime (chosen).
- Single source in TS const exported from a registry module (rejected — plugin.json is the platform-mandated manifest; making TS authoritative makes plugin.json a generated artifact and adds toolchain).
- Sync-script enforced at CI (rejected — drift still possible at runtime).

**Decision:** plugin.json is authoritative. CLI reads it at runtime via `import('./plugin.json' assert { type: 'json' })`.

**Rationale:** Closes H-3 with zero indirection.

**Consequences:** Runtime JSON parse on startup (acceptable cost). Tests cross-validate.

**Reversal Cost:** **Trivial.** <1 day if reverted.

### AVD-AD-04: extractExcellenceSpec Actually Extracts From Bundle TQCD/AVD Content (closes C-1)

**Status:** Accepted.

**Context:** v1.0.4 function takes no args, returns hardcoded strings. F13 violation by F13-prevention plugin.

**Options Considered:**
- Take ParsedBundle, extract from .tqcd.sections + .avd.sections (chosen).
- Rename function to `defaultExcellenceSpec` (rejected — that's the documentation fix, not the architectural fix; the methodology claim "backwards-plan from per-bundle excellence" demands per-bundle behavior).
- Move to template-engine pattern (rejected — over-engineering for a stable extract).

**Decision:** Function takes ParsedBundle, returns ExcellenceSpec extracted from bundle content. Hardcoded fallback only when bundle lacks the sections (with WARN log).

**Rationale:** Methodology-claim load-bearing. Closes C-1.

**Consequences:** Bundle Parser must produce structured .sections (per AVD-AC-02 contract). Tests feed real bundles; fixtures live in tests/fixtures/.

**Reversal Cost:** **Moderate.** Reverting reintroduces C-1. ~5 files including AC-02 contract.

### AVD-AD-05: A21 DRR Codification — Plugin Hook H9 + Tier 14 Hook v06 + recovery_events Audit Log Block

**Status:** Proposed (pending FM thread gate-49 follow-up rater verification per PRD-AS-03).

**Context:** Per Two-Axis Pitch §3, A21 DRR requires three layers: plugin hook (H9), commit-time enforcement (Tier 14 in commit-msg hook v06), audit-log artifact (recovery_events: block per /asae frontmatter).

**Options Considered:**
- Plugin-only (rejected — commit-time bypass possible via direct git commit).
- Hook-only (rejected — no execution-time prevention).
- All three layers composed (chosen).

**Decision:** H9 PostToolUse hook fires DRR; commit-msg hook v06 Tier 14 verifies recovery_events present when work history shows reverts; audit log records the recovery_events frontmatter as a row.

**Rationale:** Per Two-Axis Pitch §3 mechanism spec; structurally enforces DRR rather than advisory-only.

**Consequences:** v1.1.0 ships H9 behind `cdcc.experimental.drr` flag (per PRD-AS-03 fallback). Tier 14 in commit-msg hook v06 lands in a separate gate (per Two-Axis Pitch §9 sequencing).

**Reversal Cost:** **Heavy.** Three layers + the empirical claim in the v04 commercial pitch depends on this. ~10+ files. 1-3 weeks. **Note: this is the highest-stakes ADR in the bundle.** The "pending FM thread verification" status is the load-bearing flag.

### AVD-AD-08: H6 = Step-History + Cost-Telemetry Merged (vs. Split Into H6/H10)

**Status:** Accepted (provisional; revisit at v1.2.0 per PRD-OQ-01).

**Context:** Existing v1.0.4 plugin.json has H6 = step-reexec (closing A13). User merge directive: H6 = Cost Telemetry. Both are PreToolUse on Write/Edit/Bash. Splitting requires renumbering existing H6 → H10 and would create user-visible breakage.

**Options Considered:**
- Merge into H6 with combined payload (chosen).
- Renumber: keep H6 step-history, add H10 cost-telemetry (rejected — user-visible break of H6 ID semantics; also user explicitly directed merge).
- Two separate hooks both at H6 (rejected — invalid plugin.json).

**Decision:** H6 fires once per Write/Edit/Bash; emits combined audit entry with step_id + hash_of_inputs (step-history) + model + token_estimates (cost-telemetry, when Claude Cost loaded).

**Rationale:** Both share the same trigger; combining is cheaper than two hooks; user directive aligns.

**Consequences:** Schema change to audit-entry table (additive fields). Test coverage covers Cost-Cost-loaded vs absent paths (per FR-09).

**Reversal Cost:** **Moderate.** Splitting later means re-numbering H6→H10 + audit-entry schema migration. 1 week.
